# MES-RTM 项目说明

本文档基于当前项目代码整理，用于统一项目业务口径、页面职责和后续修改方向。当前项目本质上是一个基于 Vue 3 的 SMT 场景 MES 前端演示系统，核心数据仍以本地 mock 数据驱动。

## 1. 项目定位

- 项目名称：`MES-RTM`
- 项目类型：SMT 制造场景的 MES 实时制造执行前端
- 当前形态：前端单页应用，使用本地 mock 数据模拟工单、批次、工艺路线、设备、上料、进出站、维修、追溯等业务
- 主要使用场景：
  - PC 管理端
  - 车间操作终端
  - 车间看板展示

## 4. 当前业务主流程

这是当前项目已经明确下来的核心业务逻辑。

### 4.1 工单创建

先创建生产工单，主要字段包括：

- 生产产品
- 工艺路线
- 计划数量
- 交货期

工单创建后，先作为生产准备对象存在。

### 4.2 工单释放

工单创建完成后，需要执行释放。

释放的意义是：

- 工单进入可执行状态
- 才允许进入批次拆分

建议统一理解为：

- 创建工单：建立生产任务
- 释放工单：允许任务进入批次执行阶段

### 4.3 批次创建

在批次管理中，针对“已释放但尚未创建批次”的工单进行批次拆分。

一个工单可以拆分为多个批次，每个批次继承：

- 工单编号
- 产品型号
- 工艺路线
- 计划数量
- 默认产线
- 负责人

### 4.4 批次投产

批次创建后，默认状态为`待投产`。

只有执行了`投产`操作后，批次才算正式进入生产流程，状态转为`待进站`。

这一步的业务含义是：

- 批次正式进入工艺路线执行
- 未投产批次不能进入上料管理、进站操作等执行环节

### 4.5 上料管理

上料管理是批次正式进站前的准备环节。

用户可以提前对指定批次、当前工序所需物料进行补充。页面应围绕当前工序 BOM 展开，展示：

- 当前工序 BOM 完整清单
- 每个物料的应上数量
- 已上数量
- 未完成项
- 最近补料记录

上料管理的目标不是直接放行批次，而是为进站前的物料校验做准备。

### 4.6 进站操作

进站操作只允许处理已投产并处于可进站状态的批次。

当前逻辑是：

1. 选择待进站批次
2. 系统按当前工序进行 BOM 物料校验
3. 将当前批次上料信息与该工序 BOM 进行比对
4. 若物料不齐，提示用户返回上料管理补料
5. 若物料校验通过，再填写并提交：
  - 当前工序设备
  - 进站数量
  - 操作人
  - 备注

提交进站后，批次进入当前工序执行中。

### 4.7 出站操作

当前工序完成后，在出站操作中处理该批次的完工结果。

出站环节主要处理：

- 完工数量
- 不良数量
- 报废数量

如果存在不良，还需要选择不良处置方式：

- 维修
- 报废
- 强制出站

处理规则：

- 选择`维修`：应在维修管理中生成维修任务
- 选择`报废`：计入报废数据
- 选择`强制出站`：允许异常批次继续流转，但必须记录原因

出站提交后，批次流转到下一道工序，重复“上料管理 / 进站操作 / 出站操作”的流程，直到整条工艺路线全部完成。

### 4.8 维修管理

维修管理用于承接出站环节中标记为维修的不良数据。

维修管理的职责应包括：

- 查看维修任务
- 填写维修处理结果
- 区分修复、报废、关闭等处理结果
- 形成维修记录

当前项目里维修模块已经存在页面和 mock 数据，但后续如果继续深化，需要把维修后的回流逻辑再明确。

### 4.9 批次跟踪

批次跟踪本质上是批次全流程追溯视图，不只是简单日志。

应能查看某个批次从投产开始，到全部工序完成过程中的：

- 批次基本信息
- 各工序进站记录
- 各工序出站记录
- 上料记录
- 设备信息
- 数量变化
- 不良 / 报废 / 维修处理记录
- 当前状态与流转历史

## 5. 当前状态流转口径

为了后续页面统一，这里把当前批次状态口径固定下来。

### 5.1 工单层

当前工单大致可理解为：

- `draft`：草稿
- `pending`：待释放
- `running`：已释放 / 执行中
- `paused`：暂停
- `completed`：完成
- `closed`：关闭

### 5.2 批次层

当前批次状态定义为：

- `pending_launch`：待投产
- `waiting`：待进站
- `running`：生产中
- `paused`：暂停
- `repair`：维修中
- `completed`：已完成
- `locked`：已锁定

推荐按下面的主链路理解：

1. 批次创建后：`pending_launch`
2. 执行投产后：`waiting`
3. 进站并进入工序加工后：`running`
4. 当前工序异常中止：`paused` 或 `locked`
5. 不良进入维修：`repair`
6. 全部工序完成后：`completed`

## 6. 当前建议固定的数量规则

这一部分非常重要，后续页面和统计都应该按同一套规则处理。

### 6.1 出站数量关系

建议统一按下面的口径处理：

`进站数量 = 完工良品数量 + 不良数量 + 报废数量`

其中：

- 完工良品数量：正常通过当前工序的数量
- 不良数量：当前工序发现的异常数量
- 报废数量：确认不能继续生产的数量

如果后续要区分“待维修不良”和“已报废不良”，也应基于这套总量关系延展，不能各页面各算各的。

### 6.2 强制出站

强制出站不应当成为常规操作，只用于异常放行场景。

必须记录：

- 强制出站原因
- 操作人
- 操作时间

并在批次跟踪中可查。

### 6.3 维修口径

建议后续统一为：

- 出站时选择`维修`，生成维修任务
- 维修任务记录对应批次、工序、不良数量、原因
- 维修完成后，需明确结果是：
  - 修复后回流
  - 转报废
  - 关闭处理

当前项目已经有维修页面，但“维修后如何回流”这件事仍需在后续迭代中进一步落细。

负责：

- 选择待进站批次

- 提交进站

负责：

- 选择当前工序已完成的批次
- 填写完工、不良、报废数量

### 7.6 维修管理

负责：

- 查看维修任务

- 记录维修结论

负责：

- 查看批次全流程履历

## 9. 当前项目限制

当前项目仍有以下限制：

1. 主要依赖前端本地 mock 数据，刷新后不会形成真实后端持久化
2. `src/api/` 虽然已有接口封装，但页面主体仍主要直接依赖 mock 数据
3. 当前很多业务状态变更仍是前端内存态模拟
4. 维修回流闭环、工序推进闭环、批次完成判定等规则还可以继续深化
5. 部分历史文档存在编码问题，现以本文件为准

## 10. 后续建议优先级

如果继续往下改，建议优先按下面顺序推进：

1. 固化批次完整流转规则
2. 明确维修回流逻辑
3. 明确出站数量与状态变更关系
4. 补齐批次跟踪中的全过程记录
5. 再决定是否把 mock 数据逐步替换成真实接口
